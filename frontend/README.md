# frontend

Frontend MVP для кейса `Платформа ПрофДНК`.

Проект покрывает основной продуктовый сценарий из кейса:

1. админ создаёт психолога и управляет доступом
2. психолог создаёт и публикует тест
3. клиент проходит тест по публичной ссылке без регистрации
4. психолог видит результаты, метрики и формирует отчёт

## Что реализовано

### Admin

- вход администратора
- dashboard администратора
- список психологов
- создание психолога
- блокировка / разблокировка доступа
- продление подписки / доступа
- карточка психолога
- реестр подписок

### Psychologist

- вход психолога
- dashboard психолога
- профиль и карточка специалиста
- список тестов
- конструктор методик
- drag-and-drop перестановка вопросов
- formula rules для расчёта метрик
- шаблоны отчётов
- результаты по тесту
- деталка одной сессии
- генерация `HTML / DOCX` отчётов

### Public client flow

- открытие теста по публичной ссылке
- анкета респондента
- прохождение без регистрации
- сохранение прогресса
- автосохранение по паузе
- восстановление незавершённой сессии
- экран результата
- публичная карточка автора теста

## Архитектура

Основная структура:

```text
src/
  app/        # router, guards, providers, global styles
  pages/      # экраны по ролям: admin / psychologist / client / common
  modules/    # бизнес-логика по доменам: auth, admin, tests, psychologist, public-tests
  shared/     # общий api client, routes, ui primitives, utils
```

## Роли и маршруты

### Public

- `/` — авторизация
- `/session/:slug` — публичное прохождение теста
- `/session/:slug/author` — карточка автора теста
- `/result/:slug` — результат клиентской сессии

### Psychologist

- `/dashboard`
- `/tests`
- `/builder/:id`
- `/tests/:id/results`
- `/tests/:id/results/:sessionId`
- `/report-templates`
- `/profile`

### Admin

- `/admin`
- `/admin/psychologists`
- `/admin/psychologists/:id`
- `/admin/subscriptions`

## Запуск

### 1. Установить зависимости фронта

```bash
npm install
```

### 2. Поднять backend

Backend поднимается отдельно через Docker:

```bash
docker compose -f docker-compose.deploy.yml up -d
```

По умолчанию backend ожидается на:

- `http://localhost:8080`

Swagger:

- `http://localhost:8080/swagger/`

### 3. Запустить frontend

```bash
npm run dev
```

Frontend будет доступен на:

- `http://localhost:5173`

### 4. Production build

```bash
npm run build
```

## Конфигурация

Основные переменные лежат в [`.env`]

Ключевые настройки:

- `APP_IMAGE` — docker image backend
- `DB_*` — параметры PostgreSQL
- `ALLOWED_ORIGINS` — CORS origins
- `ADMIN_ACCOUNTS` — сидовые админы

Во frontend:

- в dev-режиме запросы идут через Vite proxy `/api`
- в production используется `VITE_API_URL` или fallback `http://localhost:8080`

## Как проверить основной сценарий

### Сценарий 1. Администратор

1. Зайти под админом
2. Открыть `/admin/psychologists`
3. Создать психолога
4. Проверить карточку психолога
5. Продлить доступ / подписку

### Сценарий 2. Психолог

1. Зайти под психологом
2. Открыть `/tests`
3. Создать тест
4. Перейти в `/builder/:id`
5. Добавить вопросы
6. Настроить formula rules
7. Привязать шаблон отчёта
8. Опубликовать тест

### Сценарий 3. Клиент

1. Открыть публичную ссылку теста
2. При необходимости открыть карточку автора
3. Заполнить анкету
4. Пройти тест
5. Получить экран результата

### Сценарий 4. Результаты и отчёты

1. Психолог открывает `/tests/:id/results`
2. Переходит в `/tests/:id/results/:sessionId`
3. Смотрит ответы и метрики
4. Генерирует `HTML` или `DOCX`

## Важные технические детали

### Авторизация

- backend использует `session cookies`
- для non-GET запросов нужен `X-CSRF-Token`
- frontend уже работает через cookies + csrf

### Ошибки API

Frontend ожидает единый формат:

```json
{
  "message": "Validation failed",
  "field_errors": {
    "email": "Email already exists"
  }
}
```

### Public flow

- клиент не регистрируется
- сессия привязана к публичной ссылке теста
- прогресс хранится локально и синхронизируется с backend
- при повторном открытии теста backend может вернуть незавершённую сессию

## Что зависит от backend

Полностью рабочий сценарий требует корректной работы backend-контрактов.

Особенно важно:

- стабильный payload автора теста в `GET /public/tests/{slug}`
- стабильный payload результатов и метрик
- корректная генерация `HTML / DOCX`
- стабильное обновление порядка вопросов
- единый формат `field_errors`
